# Отчёт по практике: настройка Git Hooks

## Цель работы

Цель практики — настроить локальную автоматизацию в репозитории с помощью Git Hooks, чтобы до попадания изменений в историю проекта автоматически проверялись staged-файлы, сообщение коммита, сборка проекта и быстрые unit-тесты.

## Задание 1. Базовый `pre-commit`

Файл: `.githooks/pre-commit`

В hook реализованы проверки:

- коммит останавливается, если нет подготовленных staged-изменений;
- коммит останавливается, если среди staged-файлов есть файлы с расширениями `.log` или `.tmp`;
- при ошибке выводится информативное сообщение с перечислением запрещённых файлов;
- при успешной проверке выводится сообщение о прохождении проверки.

Пример запрещённых файлов:

```text
debug.log
cache.tmp
```

Такие файлы не должны попадать в историю репозитория, потому что обычно содержат временные данные, отладочные записи или локальную информацию разработчика.

## Задание 2. Hook `commit-msg`

Файл: `.githooks/commit-msg`

В hook реализованы проверки:

- пустое сообщение коммита запрещено;
- первая значимая строка сообщения должна соответствовать формату `[TYPE] message`;
- тип пишется заглавными латинскими буквами;
- после закрывающей скобки должен быть пробел и осмысленное описание.

Примеры корректных сообщений:

```text
[FIX] null pointer crash
[FEAT] add new user API
[TEST] add calculator unit tests
```

Примеры некорректных сообщений:

```text
fix bug
[fix] bug
[FEAT]
```

Проверка пустого сообщения выполнена именно в `commit-msg`, потому что `pre-commit` запускается до создания файла сообщения коммита. Hook `commit-msg` получает путь к этому файлу первым аргументом, поэтому он является правильным местом для такой проверки.

## Задание 3. Продвинутый `pre-push`

Файл: `.githooks/pre-push`

Перед отправкой изменений на удалённый репозиторий hook запускает два шага:

1. `scripts/build.sh` — сборка проекта.
2. `scripts/test.sh` — быстрые unit-тесты.

В учебном проекте сборка реализована через `dotnet build` для небольшого C#/.NET проекта `src/CalculatorApp`. Быстрые unit-проверки запускаются через консольный тестовый проект `tests/CalculatorApp.Tests`. Дополнительно добавлен `Makefile`, чтобы структура соответствовала примеру из задания: сборку и тесты можно запускать командами `make build` и `make test`.

Если сборка или тесты завершаются с ошибкой, `pre-push` возвращает ненулевой код завершения, и Git отменяет отправку изменений. Если оба шага проходят успешно, push разрешается.

## Итог

В результате работы репозиторий получил два уровня локальной защиты:

- до коммита проверяются staged-файлы и сообщение коммита;
- до push проверяются сборка и быстрые unit-тесты.

Такая автоматизация снижает риск случайно добавить временные файлы, оставить неинформативное сообщение коммита или отправить код, который не собирается и ломает тесты.
